MVC Application design outline

Models

Models serve to carry data from one place to an other.

In many applications, Models are used in a variety of ways including partial updates, inserts and deletes.

In our application we will create a more refined architecture and create 4 different types of Models.

These four types will be named according to their utility or purpose

*Request Models
These models will be used to pass data into a Controller. These models will participate in Model Binding where data from an in-comming request will be mapped to the properties of the given Model. They will be used to perform updates, inserts or deletes. More often than not, they will contain partial list of all the properties of the entities that they represent.
*Response Models

These Models will be the objects passed out of Api Controllers. Often times these Models will be used to wrap Domain models as they are passed out of the web application.

All Response Models must inherit from BaseResponse.

*ViewModels Models

These Models will be the objects passed out of View Controllers. Often times these Models will be used to wrap Domain models as they are passed into a View

All View Models must inherit from BaseViewModel.

*Domain Models
These models will be used to most accurately represent the entities we store in our database. When returned, they should be fully hydrated with the data in the database.

These files are located here at the following path. You should inspect the files there for more information.

/models/

Controllers

Controllers act to execute the first layer of business logic in our application's request/response life cycle. They may receive a model as input parameters and will often engage 1 or more Service layer classes to achieve their tasks.

There are two type of Controllers in an MVC application

Controllers

which we will refer to as "View Controllers" because they are invoked with the intent to have an entire View or HTML page returned in the response to the browser.

All Api Controllers should inherit from BaseController

API Controllers

which are invoked usually via an Ajax request. These types of request may return HTML but that is not best practice. We will be using them exclusively to pass data to and from the web server from our HTML page.

All Api Controllers should inherit from BaseApiController

These files are located here at the following path. You should inspect the files there for more information.

/Controllers/

Views

Views are responsible for the UI of the application, the presentation. Any "business" logic that they may perform will be solely for the purpose of rendering an appropriate interface. They may receive a model, passed to it, from a View Controller.

There are serveral types of View files. The all have an extension of .cshtml

Layouts

These files are shared between the different Views of the application. You can think of them as the Template files for any given page.

These files are in most ways identical to View Files with regards to functionality and capability.

The HTML markup within these files is merged with the HTML markup generated by view files and rendered to the web browser as one file.

The layout files will be located here: /Views/Shared/

View Files

These files contain the page level specific HTML, CSS and Javascript mark up for a give page/route.

These files are located here at the following path. /Views/

_ViewStart File

This file has a number of utility/config purposes. We primarily would use it to set up the default Layout View file for the application.

THe default Layout for a view can be changed at the top the view file.

This file located at the root of /Views/

Partial View Files

These files containHTML, CSS and Javascript mark that could be used to partially render a snippet of content from within several different top level View files.

Some shops would render a Partial View as an entire response via an Ajax request. The intent would be to place this content within the existing DOM. We will NOT be using this technique because it is not very efficient.

These files can be located anywhere, but typically somewhere within /Views/
